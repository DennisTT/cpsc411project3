CPSC 411 - Stage 2: Semantic Checking

In this stage of the project we are doing the work described in Chapter 5 of the book. The goal of this stage is to produce (and test) a semantic checker for MiniJava. It accepts a parsed MiniJava program (AST) as input and verifies whether the AST is a valid Java program (i.e., a program that is ok according to the type and scoping rules of the language). 

If the program passes by the type checker it should mean that the program is "correct" in the sense that it can be compiled and run (but it doesn't, of course, make any guarantees that the program has no bugs).

Project Setup

Warning: Before installing the new files, it is advisable to make backup copy of your phase 1 project. Otherwise you run the risk of overwriting any changes you have made to some of the files that were provided in phase 1.

The zip file in which these instructions are contained is an archived copy of an Eclipse project. To import this project into Eclipse do as follows:

  1. Select "File > Import" Menu.
  2. Select "General > Existing Projects into Workspace".
  3. Select "Select Archive file" and Browse for this .zip file.
  4. Select the project and click "Finish".

The project is set up with a number of JUnit tests. The provided tests should be thought of as an executable specification. This means that you should not change any code in the test classes. Otherwise you are free to change any and all code in the project. We also encourage you to create additional tests, to more thoroughly test your solution.

Test classes who's name contains the word "Internal" are an exception to the "don't change the tests" rule. These tests are intended to test a chunk of functionality that is "internal" to your solution. You may in fact chose to change these tests. For example, in this project a file TestTypeCheckPhase1Internal.java tests the stage that builds the symbol table. As such it is dependent on specific choices of how you chose to represent the symbol table. Later stages of the compiler may (and probably will) depend on some of these details. However, we want you to explicitly think about what goes into the symbol table, and to have the freedom to choose your own implementation strategy. 

The project contains the same files from the phase 1 starter and the following new files:

  • src 
    • minijava.test.typechecker: JUnit tests for this phase of the project.
    • minijava.typechecker: Some starter code provided for you.
    • minijava.typechecker.implementation:  You should create this package. The intention is to put (as much as it is possible / reasonable) all the code implementing your type checker in this package.
    • minijava.util:  there are two classes ImpTable and FunTable that may be useful to implement symbol tables. You don't have to use them (e.g., maybe you prefer to simply use java.util.HashMap or java.util.TreeMap instead, or implement something yourself).

How to get started

Many classes in the starter code are heavily commented and provide explanation about the code. In particular, the following are good places to start:

  • The README_stage2 file at the toplevel of the src directory.
  • The files in the package minijava.typechecker.
  • The comments in the minijava.test.typechecker tests.
  • You should also read Chapter 5 of the textbook.

One further note: although you are allowed to change all code except for the tests, this does not mean that you actually have to change any code at all! In fact, you can complete this assignment by only adding code in the minijava.typechecker.implementation package.
